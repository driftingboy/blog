mysql-索引优化策略

發表於2021-06-28|更新於2021-12-14|存储
|閱讀量:

在mysql-索引文章中了解了索引的原理后，接下来，我们要讲讲索引使用的注意事项（优化策略）

索引失效

失效原因

  • 索引选择性太差
  • <>/not in 无法使用索引
  • is null 会使用索引, is not null 不会使用索引
  • where 子句违背最左匹配原则（跳过左侧索引列,直接查询右侧索引字段）
  • 后缀查询与模糊匹配btree均不支持
  • 对索引列进行计算或者使用函数
  • 索引列出现类型转换
  • Order By 需要 ‘order by 字段’或者 ‘where+order by 字段’ 能命中索引；索引失效情况如下
    • 字段与索引字段顺序, 如 idx (a,b), “…order by b,a”
    • 排序方向不同时, 如 idx (a,b), “…order by a desc,b asc”
    • 落在不同的索引上，如 idx1(a),idx2(b), “…order by a,b”
    • where条件和order by 没有同时包含索引, idx1(a),idx2(b) “…where c = 1 order by b”

查看索引使用状况

查询结果如下图

来简单讲解一下这张图的字段含义
object_type 表示统计类型
object_schema 表示统计数据库名称
object_name 表名称
index_name 表示使用的索引名称，如果为NULL表示全表扫描
count_read count_fetch 分别表示索引在计算过程中读到多少行，fetch表示最后的结果多少行
count_insert count_update count_delete 表示插入、修改、删除的次数

格外注意两点：

  1. 按照索引使用总时长排序，如果使用时长多而index_name又为null，那我们就需要查看一下到底是哪些语句导致的全表扫描。
  2. 当系统运行了一段时间，索引的使用次数仍然是0，则表示索引几乎不会被用到，可以放心删除。

索引字段的自增性

特别要注意索引字段的自增性，如果字段无序插入，那么每次插入都会重排索引树。
普通索引还好，如果是主键索引，在innodb中是聚集索引，每一次的重排会涉及到大量的数据移动，非常耗时。
