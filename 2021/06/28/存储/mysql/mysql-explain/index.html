字段详解

如下的一个表结构

当我们执行一条简单的查询语句，就会出现 object_type=simple

我们先来看看执行计划的前四个字段的含义。

  • id
    表示select的编号，id的自增顺序是select出现顺序，id=1的是驱动表。
    比如，一个复杂的查询分析结果


    中间有个NULL表示是一个计算的中间结果。

  • select_type、table
    select_type：表示查询类型， table：表示查询基于哪个表
    第一条，select_type = primary。 对应第一个select，看到primary就可以知道是一个嵌套查询了。而它的table=，表示是基于id=3的结果集进行的查询。
    第二条，select_type = derived。表示派生表，也就是跟在from后的子查询生成的一个临时表。
    第三条，select_type = union。表示 union 关键字后的查询结果集。
    第四条，select_type = union。表示 两个结果集进行union去重后得到的结果集。因为它包含一个去重计算，所以会单独产生一个结果集，看 table=<union3,4>表示来自id=3、id=4的两个结果集。
    如果将 union换成union all 则不会出现这个中间结果了，因为不需要去重计算了，只是单纯聚合。如下：

    elect_type = subquery, 包含在 select 中的子查询（不在 from 子句中）

  • partitions：说明查询作用在哪个分区表上

分析、调优sql语句，主要还是看接下来的字段。

  • type
    按照执行的效率排序，分为如下几种
    1. system, 开发时不会接触到
    2. const, 出现在 primary key 或 unique key 的所有列与常数比较时，所以表最多有一个匹配行，读取1次，速度比较快。
    3. eq_ref, 出现在 primary key 或 unique key 索引被连接使用的时候，也只会有一条记录出现
    4. ref， 出现在使用普通索引或者唯一性索引的一部分时，会产生多条记录
    5. fulltext，全文检索索引类型
    6. ref_null，一般出现在非组件、非唯一性索引，且存在null的情况下
    7. range， 范围查询，效率可高可低，看查询范围大小
    8. index， 基于索引的全表扫描
    9. all，直接全表扫描
      如果查询语句出现了all，就可以考虑是否要优化了。
  • possible_keys, key, key_len
    possible_keys: 可能使用到的索引
    key: 实际使用到的索引，如果有possible_keys，但是key为空，则出现了索引失效
    key_len: 表示索引的长度，比如长度为4，说明索引为int类型
  • ref
    查询所用到的列或常量，常见的有：const（常量），func，NULL，字段名（例：film.id）
  • rows, filtered
    rows 是mysql预估要读取的行数
    filtered 是一个百分比，一般使用上索引都很高，rows * （filtered /100）= 实际查询出的结果集。
  • Extra 这一列展示的是额外信息.

  1. distinct
    一旦mysql找到了关联的匹配行，就不再搜索了

  2. Using index
    也叫索引覆盖，在通过普通索引查询时，返回的列数据只使用了索引中的信息，避免了回表。是性能高的表现。

  3. Using where
    mysql服务器将在存储引擎检索行后再进行过滤。就是先读取整页数据，再按 where 条件进行检查，符合就留下，不符合就丢弃。

  4. using filesort
    对结果进行计算排序,可能在内存中，也可能使用临时文件（效率较差）
    对于排序,只有select 字段 与order by 字段都被索引覆盖是才允许使用Using Index

  5. using temporary
    mysql需要创建一张临时表来处理查询结果，比如去重等操作。出现这种情况一般是要进行优化的，首先是想到用索引来优化。

extend（拓展信息）

可以通过加上 extended 关键字查看索引失效的原因，但是在我在mysql:5.7中使用时报了'EXTENDED' is deprecated and will be removed in a future release.的警告，不知道后续还是否支持。
